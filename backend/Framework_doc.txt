**What is the main architecture**
This framework is a CSV-driven Fabric ETL validation framework built on `pytest`. The design is:

- `config/master.properties` decides which Fabric layers and SQL endpoints to use.
- `utils/fabric_client.py` creates authenticated connections to Fabric SQL endpoints.
- `tests/fabric/base_csv_driven_fabric_validation.py` loads CSV rows, expands them into test cases, builds queries, executes them, and applies validations.
- `utils/predefined_validations.py` contains the reusable comparison rules.
- `pytest` runs the generated test cases.
- `allure` labels and attachments make the reports readable.

So if you need one sentence for the framework:
“This is a metadata-driven ETL validation framework where test logic is generic, and the actual scenarios come from CSV plus reusable validation methods.”

**How to explain the core flow**
Use this sequence:

1. A CSV row defines a test scenario.
2. The base class reads enabled rows only.
3. It expands one row into many test cases if multiple lakehouses or tables are listed.
4. It substitutes placeholders like `{source_lakehouse}` and `{table_name}` into queries.
5. It chooses the correct Fabric client for source and target layers.
6. It executes queries in the right order.
7. It sends the results to a validation method.
8. It builds Allure labels and attachments.
9. `pytest` reports pass/fail per generated case.

**Most important methods in the base class**
In `tests/fabric/base_csv_driven_fabric_validation.py`, these are the methods to understand first:

- `setup_class`
  Initializes shared clients, validator, and test cases.
- `_load_test_cases`
  Reads CSV and keeps only enabled rows.
- `_expand_lakehouse_test_cases`
  Expands one CSV row into many concrete cases for multiple source/target lakehouses.
- `_build_dynamic_queries`
  Builds source and target SQL when the CSV gives metadata instead of full SQL.
- `_build_query_variables`
  Prepares placeholder values for query templates.
- `_build_placeholder_execution_items`
  Handles multi-table and dimension placeholder-driven execution.
- `_execute_queries_with_dynamic_order`
  Decides which query to run first, especially for recid-based validations.
- `_execute_validation`
  Main orchestration method for one test case.
- `_run_validation`
  Dispatches to the correct validation rule.
- `pytest_generate_tests`
  Dynamically creates pytest cases from CSV rows.
- `test_csv_driven_validation`
  Actual pytest test entry point.

**What feels like the newer/advanced parts of the framework**
These are the more advanced capabilities in the current design:

- Multi-lakehouse expansion from one CSV row.
- Layer-aware syntax like `BRONZE[Lakehouse]`, `SILVER[Lakehouse]`, `GOLD[Lakehouse]`.
- Placeholder-driven query generation using CSV values.
- Dynamic query order for recid-based validations.
- Metadata-driven duplicate checks using Excel metadata.
- Per-test client selection instead of one fixed source/target.
- Better reconnect/retry handling in `FabricClient`.

These are the parts I would highlight as “enhanced framework capabilities.”

**If you want to add a new validation method**
Add it in this order.

1. `utils/predefined_validations.py`
   Add the actual reusable validation method here. This is where the comparison logic should live.

2. `tests/fabric/base_csv_driven_fabric_validation.py`
   Update the validation dispatch logic so the new `validation_type` name from CSV is routed to the new method.
   The main places to check are:
   - `_run_validation`
   - any helper methods that normalize or identify validation types
   - any special-case query handling if your new validation needs only source, only target, or custom execution order

3. `data/*.csv`
   Add the new `validation_type` value in the relevant CSV rows.

4. Optional runner changes
   Usually you do not need to edit:
   - `tests/fabric/test_csv_driven_bronze_to_silver_validation.py`
   - `tests/fabric/test_csv_driven_silver_to_gold_validation.py`
   
   These are thin wrapper files. You only change them if you are creating a completely new flow with a different CSV file or different default source/target layers.

5. Optional notebook copy
   If your team also runs the notebook path, keep the duplicate Fabric notebook logic aligned:
   - `Fabric_ETL/Notebook_files/predefined_validations.py`
   - `Fabric_ETL/Notebook_files/fabric_notebook_runner.py`
   - `Fabric_ETL/fabric_client.py`

**How to think about new validation changes**
Use this split:

- Put comparison logic in `utils/predefined_validations.py`
- Put framework orchestration and CSV routing in `tests/fabric/base_csv_driven_fabric_validation.py`
- Put scenario data in CSV

That keeps the framework maintainable.

**If you want to add a new CSV file**
There are two common cases.

**Case 1: same existing flow, just another CSV**
If you want a new CSV for a different set of tests in the same flow, create:

- `data/your_new_csv_file.csv`

Then create or update a thin pytest wrapper file in `tests/fabric/` that points to that CSV.

Example:

- `tests/fabric/test_csv_driven_silver_to_gold_validation.py`
- `tests/fabric/test_csv_driven_bronze_to_silver_validation.py`

These wrapper files mainly control:

- `CSV_FILE`
- `SOURCE_LAYER`
- `TARGET_LAYER`

So if you add a new CSV, the usual change is:

- create a new wrapper file or copy an existing one
- change only the class attributes

For example, the wrapper normally looks like:

- `CSV_FILE = "data/your_new_csv_file.csv"`
- `SOURCE_LAYER = "SILVER"`
- `TARGET_LAYER = "GOLD"`

**Case 2: completely new layer flow**
If you want a new path such as AX -> Bronze or another custom Fabric flow, then edit or create:

1. `data/your_new_flow_tests.csv`
2. `tests/fabric/test_csv_driven_your_new_flow_validation.py`
3. `config/master.properties` only if the new flow needs new endpoints or layer configuration
4. `tests/fabric/base_csv_driven_fabric_validation.py` only if the new flow needs new orchestration behavior beyond just changing layers

**Minimum files to edit for a new CSV file**
In most cases, only these are needed:

1. `data/new_file.csv`
2. one wrapper file under `tests/fabric/`

Usually no change is needed in:

- `utils/predefined_validations.py`
- `utils/fabric_client.py`
- `tests/fabric/base_csv_driven_fabric_validation.py`

unless the new CSV introduces a new validation type or new execution behavior.

**Quick checklist for adding a new CSV**

1. Create the CSV with the same header structure as an existing working file.
2. Add test rows with `enabled = TRUE` only for the cases you want active.
3. Create or update a wrapper test file in `tests/fabric/`.
4. Set:
   - `CSV_FILE`
   - `SOURCE_LAYER`
   - `TARGET_LAYER`
5. Run pytest against that wrapper file.

**Quick checklist for adding a new validation**

1. Add method in `utils/predefined_validations.py`
2. Route it from `tests/fabric/base_csv_driven_fabric_validation.py`
3. Add `validation_type` in CSV
4. Run one focused test row first
5. Then run the full wrapper file
